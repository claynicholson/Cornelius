Cornelius Documentation

Cornelius is a Hack Club YSWS (You Ship We Ship) submission review engine. It automatically reviews GitHub repositories for project completeness, code quality, and authenticity using a combination of rule-based checks and AI-powered analysis via Claude.

Whether you are reviewing hardware projects with PCB designs and 3D models, or software projects with deployed URLs, Cornelius provides a structured, reproducible review process with configurable presets and a trust score system.

Base URL
http://localhost:3000

# Quick Start

1. Install

# Clone and install
git clone https://github.com/your-org/cornelius.git
cd cornelius
npm install
npm run build

2. Configure Environment

# .env file
GH_PROXY_API_KEY=your-hackclub-gh-proxy-key   # Required
ANTHROPIC_API_KEY=sk-ant-...                 # Optional: enables AI checks
PORT=3000                                    # Optional: default 3000
HC_CLIENT_ID=...                              # Optional: Hack Club OAuth
HC_CLIENT_SECRET=...                          # Optional: Hack Club OAuth
SESSION_SECRET=...                            # Optional: session encryption

3. Run Your First Review

# CLI
cornelius review https://github.com/user/project -p software-shipped

# Or start the web server
npm run server

# Then via API
curl -X POST http://localhost:3000/api/review \
  -H "Content-Type: application/json" \
  -d '{"url": "https://github.com/user/project", "preset": "default"}'

# Architecture

Cornelius has three main components:

  • Core Review Engine -- Orchestrates checks against a repo context (file tree, README content, metadata). Each check returns a structured CheckResult.
  • AI Layer (Claude) -- Optional integration with Anthropic's Claude for deep analysis: README quality, code review, website inspection, effort verification. Includes budget controls.
  • Presets -- JSON configuration files that define which checks run, whether they are required, and their parameters. Different presets for hardware vs software projects.

The system is accessible via three interfaces: a web dashboard (Express), a REST API, and a CLI. All three use the same core reviewer engine.

Checks Reference

Each review runs a configurable set of checks. Checks are either rule-based (deterministic, no API cost) or AI-powered (uses Claude, costs tokens).

▼ github_link_works RULE

Validates that the GitHub URL is reachable and the repository exists with files.

Pass: Repository exists, is accessible, and contains at least one file.
Fail: Repository is 404, private, or completely empty.
Config: None. Always runs first.
▼ readme_present RULE

Checks for a README file at the repository root.

Pass: Found README.md, README, README.txt, README.rst, or README.adoc at repo root.
Fail: No README file found.
Config: None.
▼ readme_quality AI

Evaluates README content for project description, build/setup instructions, and overall quality. Uses Claude AI when available; falls back to a heuristic based on word count and heading count.

Pass: README has meaningful project description, setup instructions, and decent length.
Fail/Warning: README is too short, lacks description, or is just a template.
Config: severity -- "fail" or "warning" (default: "warning" in most presets).
▼ readme_has_project_image AI

Detects Markdown/HTML image tags in the README. Optionally uses AI to determine if images are project-relevant (hardware photos, 3D renders, PCB layouts, screenshots).

Pass: Found at least minimum_image_count project-relevant images.
Fail: No images or only irrelevant images (badges, logos).
Config: minimum_image_count -- number (default: 1).
▼ three_d_files_present RULE

Scans the repository file tree for 3D design / CAD files. Supports STL, STEP, 3MF, OBJ, Fusion 360, FreeCAD, and OpenSCAD formats.

Pass: Found at least minimum_file_count files with matching extensions.
Fail: No 3D files found (or below minimum).
Config: minimum_file_count -- number (default: 1), allowed_extensions -- comma-separated list (default: ".stl,.step,.stp,.3mf,.obj,.f3d,.fcstd,.scad").
▼ pcb_files_present RULE

Detects PCB design files from KiCad, EasyEDA, Eagle, Altium, and Gerber outputs.

Pass: Found source design files from a recognized ECAD tool.
Fail: No PCB files found, or only Gerbers with no source design files (when required).
Config: allowed_ecad_tools -- comma-separated list (default: "kicad,easyeda,eagle,altium"), require_source_design_files -- boolean (default: true).
▼ bom_present_if_required RULE

Looks for a Bill of Materials file (bom.csv, parts.csv, components.csv, etc.) or a BOM section in the README.

Pass: Found BOM file or BOM section in README.
Fail/Warning: No BOM found.
Config: severity -- "fail" or "warning", minimum_columns -- array of required columns (e.g. ["part", "quantity"]).
▼ source_code_present RULE

Checks that the repository contains actual source code files.

Pass: Found at least minimum_file_count source files (.js, .ts, .py, .rs, .go, .java, etc.).
Fail: Fewer than the minimum source files found.
Config: minimum_file_count -- number (default: 3).
▼ gitignore_present RULE

Checks for a .gitignore file to prevent committing build artifacts and dependencies.

Pass: .gitignore found at repo root.
Fail: No .gitignore file.
Config: None.
▼ package_manager_present RULE

Detects package manager / dependency files: package.json, requirements.txt, Cargo.toml, go.mod, pom.xml, build.gradle, Gemfile, etc.

Pass: At least one recognized dependency manifest found.
Fail/Warning: No dependency file found.
Config: severity -- "fail" or "warning".
▼ license_present RULE

Checks for a LICENSE or COPYING file in the repository.

Pass: LICENSE, LICENSE.md, COPYING, or similar found.
Fail: No license file found.
Config: None.
▼ code_quality_overview AI

AI assessment of code structure, effort, and originality. Samples key files from the repository and asks Claude to evaluate overall quality.

Pass: Code shows reasonable structure, effort, and is not purely boilerplate.
Fail/Warning: Code is minimal, entirely generated, or just a starter template.
Config: severity -- "fail" or "warning".
▼ url_alive AI

Checks that a deployed website / playable URL is accessible and serves a real application. Uses AI to verify the response is meaningful, not a parking page or error.

Pass: URL responds with a real application or content.
Fail: URL is unreachable, returns errors, or is a placeholder.
Config: Pass playableUrl in the review request body (or -u in CLI).
▼ deep_code_review AI

Deep AI-powered code review that assesses originality, shipped status, and code quality. Reads multiple source files and analyzes patterns, effort, and authenticity. Budget: approximately $0.50 per review.

Pass: Code shows genuine effort, originality, and is a real shipped project.
Fail: Code is copied, trivial, or not a real project.
Config: severity -- "fail" or "warning". Controlled by preset maxBudget.

Planned / Upcoming Checks

These checks are in development or planned for future releases:

▼ website_mole AI NEW

Deep website inspection. Crawls the deployed URL and analyzes page structure, interactivity, and whether the site demonstrates real functionality beyond a static landing page.

Pass: Website shows genuine interactive content, multiple pages, or real application logic.
Fail: Single static page, template with no customization, or broken deployment.
▼ git_forensics AI NEW

Git history analysis. Examines commit patterns, authorship, timing, and development cadence to assess whether the project was genuinely developed over time or bulk-committed.

Pass: Commit history shows organic development with multiple meaningful commits over time.
Fail: Single bulk commit, suspicious commit patterns, or mismatched authorship.
▼ repo_metadata RULE NEW

Repository metadata analysis. Checks creation date, fork status, star count, description, topics, and other GitHub metadata signals.

Pass: Repository has a description, is not a fork (or fork is justified), and has reasonable metadata.
Fail: Repository is a direct fork with no modifications, or has suspicious metadata.
▼ template_detection RULE NEW

Detects if the repository was created from a starter template (e.g. create-react-app, Next.js starter, Rails scaffold) with minimal modification.

Pass: Project shows significant modifications beyond the template.
Fail: Project is essentially an unmodified starter template.
▼ ai_code_detection AI NEW

Detects AI-generated code patterns. Analyzes code style, comments, naming conventions, and structure for signs of AI generation (ChatGPT, Copilot, Claude, etc.).

Pass: Code appears to be human-written or shows human modification of AI-assisted code.
Fail: Code is predominantly AI-generated with little human involvement.
▼ effort_verification AI NEW

Cross-verifies self-reported hours against repository evidence. Compares journal entries, commit history, code complexity, and project scope to validate hour claims.

Pass: Reported hours are consistent with observable evidence.
Fail: Major discrepancy between reported hours and project scope / commit history.

Trust Score System

The trust score is a composite metric (0.0 to 1.0) derived from individual check results. It represents overall confidence in the submission's legitimacy and completeness.

How It Works

The confidenceScore returned in every review is the average confidence across all non-skipped checks. Each individual check reports its own confidence (0.0 to 1.0) based on how certain it is of its result.

Score Thresholds

RangeMeaningVisual
0.80 - 1.00 High confidence. Checks are conclusive.
0.50 - 0.79 Moderate confidence. Some checks inconclusive or AI-dependent.
0.00 - 0.49 Low confidence. Many checks failed or errored. Manual review recommended.

Category Breakdown

Trust signals come from these categories of checks:

CategoryChecksMeasures
Code Authenticity deep_code_review, code_quality_overview, ai_code_detection Is the code original and human-written?
Effort Verification effort_verification, git_forensics Does commit history and scope match claimed effort?
Project Completeness readme_present, readme_quality, source_code_present, bom_present, 3d/pcb files Does the project have all expected deliverables?
Deployment Status url_alive, website_mole Is the project actually deployed and functional?
Development Process gitignore_present, package_manager, license_present, template_detection Does the project follow good development practices?

Flag Severities

Each check can produce results at different severity levels:

  • pass -- Check passed. No issues found.
  • warning -- Non-blocking issue. Reviewer should note but does not fail the submission.
  • fail -- Blocking issue (if check is required). Causes overall review to fail.
  • error -- Check could not run (API error, timeout, etc.).
  • skipped -- Check is disabled in the current preset.

Presets Guide

Presets configure which checks are enabled, whether they are required, and their parameters. Choose the right preset for the type of project you are reviewing.

Built-in Presets

default
hardware
Standard hardware review. BOM and README quality are warnings only. Good starting point for most hardware YSWS reviews.
hardware-default
hardware
Strict hardware review. All checks required including BOM, README quality, and project images. For programs that require complete hardware documentation.
software-default
software
Standard software review. Disables hardware checks (3D, PCB, BOM). Enables source code, gitignore, package manager, and code quality as warnings.
software-shipped
software
Strict software review for shipped projects. Requires deployed URL alive, deep code review, README quality. Budget: $0.55 per review.
blueprint
hardware
Hack Club Blueprint program preset. All hardware checks required with broad 3D format support. BOM must include part and quantity columns.

Additional presets you can create for your needs:

  • software-strict -- All software checks required with fail severity.
  • hardware-strict -- All hardware checks required with strict validation.
  • quick-check -- Only rule-based checks, no AI. Fast and free.
  • hackathon -- Relaxed checks appropriate for hackathon submissions (most checks as warnings).

Custom Presets

Authenticated users can create, update, and delete custom presets via the API. Custom presets are stored per-user and take priority over built-in presets with the same name.

Preset JSON Schema

{
  "name": "My Custom Preset",
  "projectType": "software",        // "hardware" or "software"
  "instructions": "software-shipped", // instruction template name (optional)
  "maxBudget": 0.50,                  // max AI cost in USD (optional)
  "checks": {
    "github_link_works": {
      "enabled": true,
      "required": true
    },
    "readme_quality": {
      "enabled": true,
      "required": false,
      "severity": "warning"       // "fail" or "warning"
    },
    "source_code_present": {
      "enabled": true,
      "required": true,
      "minimum_file_count": 5       // check-specific config
    }
    // ... other checks
  }
}

CSV Format

The batch review endpoint and CLI accept CSV files with the following columns:

ColumnRequiredDescription
github_url REQUIRED GitHub repository URL
submission_id optional Unique submission identifier
participant_name optional Name of the submitter
email optional Submitter email
project_type optional Override project type ("hardware" or "software")
program_preset optional Override preset name per row
hours_reported optional Self-reported hours spent
journal_count optional Number of journal entries
journal optional Journal text content
playable_url optional Deployed website URL for url_alive check
notes optional Reviewer notes (not used by engine)

Example CSV

github_url,submission_id,participant_name,hours_reported,playable_url
https://github.com/alice/my-keyboard,001,Alice,25,
https://github.com/bob/web-app,002,Bob,40,https://bobs-app.vercel.app
https://github.com/carol/robot-arm,003,Carol,60,

CLI Usage

    
Cornelius provides a command-line interface for reviews and batch processing.


    
    

      

        ▼
        cornelius review <url>
      

      

        
Review a single GitHub repository from the command line.



cornelius review <github-url> [options]

Options:
  -p, --preset <name>    Review preset to use (default: "default")
  -u, --url <url>        Playable URL for deployed site verification
  --no-ai                Disable AI-powered checks


        
Examples


# Basic hardware review
cornelius review https://github.com/alice/keyboard-project

# Software review with deployed URL
cornelius review https://github.com/bob/web-app -p software-shipped -u https://bobs-app.vercel.app

# Quick check without AI (free, fast)
cornelius review https://github.com/carol/project --no-ai

# Blueprint preset for hardware
cornelius review https://github.com/dave/robot -p blueprint


        
Exit code: 0 if review passes, 1 if it fails.

      

    


    
    

      

        ▼
        cornelius batch <csvFile>
      

      

        
Review a batch of repositories from a CSV file.



cornelius batch <csv-file> [options]

Options:
  -p, --preset <name>       Review preset (default: "default")
  -o, --output <file>       Output file path (default: "results.csv")
  -f, --format <type>       Output format: "csv" or "json" (default: "csv")
  -c, --concurrency <n>    Concurrent reviews (default: 5)
  --workers                 Force worker threads for parallelism
  --no-workers              Single-threaded mode
  --no-ai                   Disable AI-powered checks


        
Examples


# Standard batch review
cornelius batch submissions.csv -o results.csv

# JSON output with high concurrency
cornelius batch submissions.csv -f json -o results.json -c 10

# Software preset, single-threaded
cornelius batch submissions.csv -p software-shipped --no-workers
